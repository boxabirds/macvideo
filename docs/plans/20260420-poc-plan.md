**Last updated:** 2026-04-20

**Maintenance instructions:**

- Record actual results under each POC's `Outcome` block when complete: what passed, what didn't, what surprised you, and what it changes in the main plan (`20260420-initial-prototyping.md`).

---

## Principles

- **Each POC is independently testable.** One script, one input, one output, one binary pass/fail. No POC should depend on another's scaffolding beyond the shared `pyproject.toml`.
- **Risk-ordered.** The unknowns most likely to kill the architecture run first. Cheap known-good stages (ffmpeg, librosa) don't need POCs — validate in integration.
- **Cheap inputs.** Use short audio clips, single stills, minimal prompts. Model downloads dominate wall-time — structure POCs so the first one warms the cache for the rest.
- **Review gates.** After each POC, decide: continue, pivot, or rethink. Don't chain POCs blindly.

## POC layout on disk


Each POC directory is self-contained: its own README stating goal + pass criteria, its own script, its own inputs, its outputs (gitignored), and a `RESULT.md` written after running to record what happened.

## POC 1 — LTX-2.3 smoke test on M5 Max

**Goal:** Prove `mlx-video` installs, downloads weights, and generates one clip on the user's hardware.

**Risk addressed:** Everything downstream dies if this fails. Guide's benchmarks and resolution claims are untested on M5 Max.

- [ ] Install `mlx-video` pinned to a resolved commit SHA
- [ ] Run text-to-video at `poc` resolution (512×320), distilled pipeline, hardcoded prompt, no audio
- [ ] Measure: download time (one-off), generation wall-time, peak memory
- [ ] Compare `mlx_video.ltx_2.generate --help` output against the flag list in `docs/20260420-music-video-ltx23-mac.md` §4. Record any drift in RESULT.md.

**Pass:** Clip plays in QuickLook, content roughly matches prompt, wall-time within 2× of the guide's "6–12 min" estimate for this resolution.

**If it fails:** Project is architecturally blocked. Fall back to cloud video gen (reconsider the no-cloud decision) or park the project.

## POC 2 — LTX-2.3 audio conditioning actually does something

**Goal:** Verify that supplying `--audio-file` actually changes motion vs. not supplying it, and that *different* audio produces *different* motion.

**Risk addressed:** The whole "music drives visuals" premise. If audio conditioning is weak or ignored, we plan clips on timing alone and drop the premise.

- [ ] Pick one prompt (e.g. "slow drift across a dark ocean surface, moonlight, 16mm grain").
- [ ] Generate three clips at `poc` resolution with the same seed:
  - (a) no audio conditioning
  - (b) `--audio-file` = quiet drone (5s of ambient pad)
  - (c) `--audio-file` = driving beat (5s of techno)
- [ ] Play all three side by side.

**Pass:** (b) and (c) visibly differ in motion energy or pacing, and both differ from (a). If all three look identical, audio conditioning is effectively inert.

**If it fails (weak/absent):** Drop the `--audio-file` stage from the pipeline. Rely on LLM-planned timing and NLE beat-snapping. Not fatal.

## POC 3 — LTX-2.3 image-to-video from a static still

**Goal:** Verify I2V mode works — a supplied still becomes frame 0, motion unfolds from there.

**Risk addressed:** The keyframe-first architecture (Gemini still → LTX animate) depends entirely on I2V quality.

- [ ] Generate or pick one still (can use POC 5's output if that's already run, otherwise any PNG).
- [ ] Run LTX-2.3 in I2V mode at `poc` resolution with a prompt describing *motion and change* (per docs.ltx.video guidance).
- [ ] Inspect: does frame 0 match the still? Does motion make sense from that starting point? Any style drift?

**Pass:** Frame 0 matches the input still within rounding/codec tolerance. Motion unfolds naturally. No wild style mutation by frame 24.

**If it fails:** Still-first architecture is dead. Back to text-to-video with no still control, which means giving up identity consistency.

**Gotcha:** Requires `mlx-video` to expose an `--init-image` / `--first-frame` flag. If absent, POC fails and we file a feature request / patch mlx-video ourselves.

## POC 4 — LTX-2.3 chained shots (last-frame → first-frame)

**Goal:** Generate two consecutive clips where clip B's first frame is clip A's last frame, producing frame-exact continuity.

**Risk addressed:** "Held shot" / "sustained push" sections of the shot plan. Without this, every cut is a hard cut even where we want continuity.

- [ ] Generate clip A at `poc` resolution.
- [ ] Extract clip A's last frame (ffmpeg).
- [ ] Generate clip B using that frame as the first-frame condition and a prompt describing motion continuing from A.
- [ ] Concat A + B with ffmpeg. Watch the join frame-by-frame.

**Pass:** No visible jump, flash, or style mutation at the seam.

**If it fails:** Drop `chain_from_previous: true` from the shot-plan schema. All cuts become hard cuts. Not fatal — we still get a working pipeline, just a less sophisticated one.

## POC 5 — Gemini `gemini-3.1-flash-image-preview` single still

**Goal:** Prove API access, auth, and prompt adherence for a single still. Confirm the preview model ID still resolves.

**Risk addressed:** Preview model IDs rotate. Auth/quota/billing surprises. Image quality at our target aesthetic.

- [ ] Set up `GEMINI_API_KEY` in env, document in POC README.
- [ ] Generate one still from a prompt in one of the configured styles (e.g. "steampunk, brass gears, gaslamp glow, volumetric steam").
- [ ] Log: model ID used, cost, latency, output resolution.

**Pass:** Image returns, matches prompt, quality acceptable for an LTX-2.3 seed frame. Cost per image is bounded and acceptable.

**If it fails (model ID not found):** Try `gemini-2.5-flash-image` / other current IDs. Update `20260420-initial-prototyping.md` with the working ID.

## POC 6 — Gemini identity consistency across 5 stills

**Goal:** Generate 5 stills of the same performer in 5 different scenes, passing prior stills as reference. Verify the performer actually looks like the same person.

**Risk addressed:** The entire rationale for choosing Gemini over Flux+IP-Adapter. If identity drifts, we wasted the switch.

- [ ] Generate still 1 from scratch: "A weathered mariner, mid-50s, grey beard, sou'wester, on a fishing boat".
- [ ] Generate stills 2–5 using stills 1..(n-1) as reference images, varying the scene each time (harbour, pub, cliff, rain-lashed deck).
- [ ] Show all 5 side-by-side. Would an audience read this as the same person?

**Pass:** Identity holds across all 5 shots. Some wardrobe variation is fine; the face and build should be stable.

**If it fails (drift by shot 3+):** Choices: accept drift (visible-but-tolerable), cap hero-shot count hard (≤3 shots per performer), or revert to Flux+IP-Adapter. Update the main plan either way.

## POC 7 — WhisperX word-level timings on real vocals

**Goal:** Transcribe a 30–60s vocal stem from one of the user's actual tracks and verify word timings are accurate enough to drive shot cuts.

**Risk addressed:** Transcription quality on processed vocals. If timings are ±300ms, auto-cutting on words will feel sloppy.

- [ ] Pick one user track. Demucs-separate vocals stem.
- [ ] Run WhisperX large-v3 on the vocals stem.
- [ ] Spot-check 10 words across the excerpt: are the start/end timings within ±100ms by ear?
- [ ] Record word-accuracy %: how many words did Whisper get right?

**Pass:** ≥85% word accuracy, ≥80% of spot-checked timings within ±100ms.

**If it fails:** Document which track characteristics break it (heavy reverb, autotune, vocoder). Add a manual-correction step to stage 2 or skip lyric-driven cutting for those tracks.

## POC 8 — LLM shot planner on a real song

**Goal:** Given real lyrics (from POC 7) + a styles list, have `gemini-3-flash-preview` produce a shot plan YAML. User reviews by reading, not generating.

**Risk addressed:** The director prompt template. If the LLM-generated shot plan reads as incoherent or generic, the whole pipeline output will.

- [ ] Use lyrics + timings from POC 7.
- [ ] Call Gemini with the three-step prompt: (a) pick style, (b) expand style to visual cues, (c) write shot plan as paragraphs per shot per the LTX-2.3 six-element structure.
- [ ] Output: `pocs/08-shot-plan/outputs/<track>.yaml`.
- [ ] **Review by eye** — no generation runs from this POC's output.

**Pass:** User reads the shot plan and says "yes, this is a reasonable music video for this song." Prompts are 4–8 sentences, present tense, ≤200 words each, use documented camera vocabulary, follow the chosen style consistently.

**If it fails:** Iterate on the prompt template. Cheap — pure LLM cost. Run until the planner output is usable.

## POC 9 — End-to-end 30-second integration

**Goal:** Take a 30-second section of a real track, run the full pipeline at `poc` resolution, produce a rough cut.

**Risk addressed:** Integration issues that don't show up in isolated POCs — data-shape mismatches between stages, timing drift in assembly, audio-video sync at concat.

- [ ] Pick a 30s excerpt of a user track (ideally a verse or chorus with 3–5 lyric lines).
- [ ] Run stages 1 → 6 end to end: Demucs → WhisperX → structure → shot plan → Gemini keyframes → LTX-2.3 clips → ffmpeg assembly.
- [ ] Use POCs 1–8's validated components.

**Pass:** 30s rough cut plays cleanly, audio stays in sync throughout, clips roughly match the LLM's plan for each section.

**If it fails:** Diagnose which stage's output didn't cleanly feed the next. Most likely: shot-plan YAML schema vs. generator script expectations, or audio-slice boundary rounding.

## Parallelisation

Three independent tracks run in parallel (different machines or different times of day on the same one):

- **LTX track (POCs 1 → 2/3/4):** depends on M5 Max compute. Run first because model download dominates.
- **Gemini track (POCs 5 → 6):** API-only, no compute constraint.
- **Text track (POCs 7 → 8):** CPU / modest GPU for Whisper. Run anytime.

POC 9 blocks on everything.

## Which POCs can kill the project

- **POC 1 fail** → LTX-2.3 doesn't run on Mac as claimed. Existential.
- **POC 6 fail** → Gemini identity drifts badly. Survivable but dents the "one performer, multiple scenes" ambition — re-evaluate Flux+IP-Adapter.
- **POC 7 fail on all tracks** → transcription unreliable, lyric-driven cutting is off the table.

Everything else is survivable with graceful degradation.
